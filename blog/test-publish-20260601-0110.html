Why shake feels harder than tap

A tap is a single binary event. The phone screen reports one touch, one timestamp, one location, and the game decides if the player won or lost the round. A shake is none of those things. A shake is a noisy waveform from a three-axis accelerometer that has to be filtered, thresholded, and timed, all while the user is laughing and waving a phone around a kitchen with friends watching. The honest version of the game has to handle that mess without feeling unfair, because nothing kills a party game faster than a player swearing the round was rigged.

The mechanic that ships is a windowed peak detector. The phone reads acceleration on x, y, and z about a hundred times a second. The game subtracts gravity, computes a magnitude, and watches for the moment that magnitude crosses a threshold and stays above it long enough to count as a real shake instead of a phone bump or a sneeze. The threshold is tuned per device because a budget Android handset and a flagship iPhone do not report the same numbers for the same physical motion, and a single global value would lock out half the room. Calibration runs once on first launch and then quietly recalibrates on every fresh game so a sleepy phone in a warm pocket does not drift out of spec.

What the scoring loop does

Once a shake is detected, the scoring loop has two jobs. The first is to credit the shake to the correct player without letting a slow finger on the pass-the-phone hand-off steal points. The second is to pace the rounds so that the difficulty ramps in a way that feels earned rather than punitive. Both jobs sound trivial in a design doc and stop being trivial the moment you put four loud people in a room with a single phone and a half-finished bottle on the table.

The fix for the hand-off problem is a short lockout window after every detected shake. For about two hundred milliseconds after a successful shake, the detector ignores new peaks. That window is long enough to let the phone settle between players and short enough that a fast solo round still feels responsive. The ramp problem gets solved by tying the threshold to the round number rather than to a fixed timer. Round five demands a sharper peak than round one. Players feel the game getting harder because the game is getting harder, not because the random seed turned against them.

How a round comes together

A finished round is the sum of four small systems firing in order. None of them are clever on their own. The trick is that all four have to agree before a score gets written, otherwise a single miscount blows the trust on the whole game.

  • A motion buffer collects raw accelerometer samples on a rolling window.
  • A peak detector marks the candidate shake and stamps it with a phone-local timestamp.
  • A debouncer suppresses any follow-up peaks inside the lockout window.
  • A scorer awards the point, advances the round counter, and bumps the difficulty curve.

The order matters. A scorer that fires before the debouncer would double-count a single shake. A debouncer that fires before the peak detector would swallow real motion. The system reads obvious in hindsight and was painful to get right the first time. You can try the result yourself at app.shakegasm.com if the curiosity overruns the patience.

How the test post got here

This paragraph exists because the publish pipeline has a word floor and the test post needed to clear it without padding. The pipeline runs an end-to-end check: a draft passes the SEO spine validator, the create call writes an HTML file into the landing site repo, the commit pushes to main, Cloudflare Pages picks the push up, the build finishes, and the live URL serves a two hundred response. After that, the delete call removes the post, the index regenerates, the sitemap drops the entry, and the cleanup commit lands on the same branch. The whole loop should take under three minutes on a good day and under six on a bad one. The report at the end says which one this was.

What a green run proves

A green run proves four things at once. First, the MCP server is reachable and the auth token still works. Second, the SEO spine validator agrees that the draft clears the hard rules, which means the next real post written to the same shape will also clear them. Third, the Cloudflare Pages project is still wired to the right branch of the right repo, which is the failure mode that scares Roy the most because it tends to break silently. Fourth, the delete path works and does not leave orphan files behind in the repo or the sitemap. If any of those four checks fail, the report calls it out by name and the next run starts from a known position instead of a vibe. The real game lives at app.shakegasm.com.
